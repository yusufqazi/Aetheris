# Aetheris

Aetheris is a polished full-stack research workspace for pharma and clinical document intelligence. It demonstrates multi-agent orchestration, PDF ingestion, chunked retrieval, structured reasoning, debate-based consensus, and report generation in a Vercel-ready Next.js application.

Important: this app is for educational and research support only. It must not be used as a diagnostic, prescribing, or treatment system, and its outputs must not replace physician judgment, regulatory review, or formal medical decision-making.

## Product Overview

Aetheris lets a user:

1. Sign in with Supabase magic-link auth or continue in demo mode
2. Upload one or more clinical or pharma PDFs
3. Ask a research question
4. Run a multi-agent analysis workflow
5. Inspect each agent's output separately
6. Review a debate/consensus layer
7. Copy or download the final report
8. Reopen saved research sessions from the dashboard

## Multi-Agent Workflow

```text
User uploads PDFs
  -> /api/upload extracts PDF text
  -> documents are chunked into retrieval-ready segments
  -> question-aware ranking selects the most relevant chunks
  -> Literature Search Agent identifies best supporting excerpts
  -> Drug Interaction Agent reviews co-administration concerns
  -> Adverse Reaction Agent extracts safety signals
  -> Clinical Trial Summarizer Agent condenses study design and outcomes
  -> Debate / Consensus Agent compares agent conclusions
  -> Report Generation Agent assembles the final structured briefing
  -> session is returned to the UI and can be persisted to Supabase
```

## Architecture

### Frontend

- Next.js App Router
- TypeScript
- Tailwind CSS
- Responsive enterprise dashboard UI

### Backend

- Route handlers under `app/api/*`
- PDF extraction with `pdf-parse`
- Chunked retrieval layer designed to support future vector search
- Optional OpenAI structured JSON generation
- Demo-mode fallback when API keys are missing

### Persistence

- Supabase-ready session persistence
- Browser local storage fallback in demo mode
- Optional SQL schema included under [`supabase/schema.sql`](./supabase/schema.sql)

## Environment Variables

Create `.env.local` from [`.env.example`](./.env.example).

```bash
OPENAI_API_KEY=
OPENAI_MODEL=gpt-4.1-mini
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
SUPABASE_SERVICE_ROLE_KEY=
```

Notes:
- If `OPENAI_API_KEY` is missing, the app falls back to deterministic demo outputs.
- If Supabase keys are missing, the app still works in demo mode using local storage.

## Local Setup

```bash
npm install
npm run dev
```

Open [http://localhost:3000](http://localhost:3000).

## Supabase Setup

1. Create a Supabase project.
2. Run the SQL in [`supabase/schema.sql`](./supabase/schema.sql).
3. Add the project URL, anon key, and service-role key to `.env.local`.
4. Configure row-level security policies appropriate for your auth model.

## Vercel Deployment

This project is Vercel-ready.

1. Import the repo into Vercel.
2. Add the environment variables from `.env.example`.
3. Deploy.

## Current MVP Notes

- Retrieval is implemented with a clean chunk-ranking layer instead of full vector embeddings.
- The architecture is intentionally prepared for a future embedding store or Supabase pgvector upgrade.
- Agent toggles are respected, and disabled agents return explicit "not run" states.
